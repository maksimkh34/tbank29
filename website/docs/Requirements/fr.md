# Скоуп требований

**FinAssist** — мобильное приложения для учета личных финансов с серверной частью.

- В **MVP** входят авторизация (с регистрацией), несколько интеграций, категоризация транзакций, ручное редактирование категорий транзакций, самих транзакций и бюджетов, отдельные счета с балансами, базовая аналитика и ИИ-ассистент.

- Вне **MVP**: алгоритмы ML, продвинутые интеграции, расширенная аналитика, экспорт отчетов.

# Функциональные требования

### Регистрация

- **ID**: UC-001
- **Название**: Регистрация пользователей 
- **Акторы**: пользователь (основной), бекенд (вторичный)
- **Предусловия**: 
    - Приложение установлено.
    - У приложения есть доступ в интернет.
    - Пользователь не авторизован.
- **Основной поток:**
    - Пользователь выбирает действие "**Зарегистрироваться**"
    - Приложение отображает форму ввода данных: ник, почту, пароль
    - Пользователь вводит данные в форму и завершает ввод данных
    - Система валидирует данные и возвращает токен авторизации
    - Приложение принимает ответ, запоминает токен и отображает успешность авторизации
    - Приложение предлагает включить синхронизацию
- **Постусловия:**
    - У пользователя есть аккаунт на сервере, у приложения есть авторизационный и обновляющий токен.
- **Альтернативные потоки:**
    - **А1**. Сервер вернул ответ о невалидных данных
        - Приложение уведомляет пользователя о некорректности какого-то поля и показывает требования к нему без ввода данных заново.
    - **А2**. Сервер вернул ответ о неподходящих данных (например, почта уже зарегистрирована)
        - Приложение уведомляет пользователя о проблеме с данными и рекомендует действия по исправлению.
- **Исключительные потоки:**
    - **E1**. Ошибка до момента отправления пакета (например, в формировании запроса).
    - **E2**. Тестовый запрос не дошел (плохая связь, сервер не работает).
        - Приложение уведомляет пользователя о работе в оффлайн режиме и предлагает создать локальную учетную запись с последующей привязкой к аккаунту (для **E1** и **E2**, для пользователя нет разницы в этих двух потоках)
- **Бизнес-правила:**
    - **BR-001**. Система может работать в оффлайн- и онлайн-режимах. Для онлайн-режима требуется активная учетная запись FinAssist на устройстве пользователя.

### Вход в систему

- **ID**: UC-002
- **Название**: Вход в систему
- **Акторы**: пользователь (основной), бекенд (вторичный)
- **Предусловия**: 
    - Приложение установлено.
    - У приложения есть доступ в интернет.
    - Пользователь не авторизован.
    - У пользователя есть аккаунт.
- **Основной поток:**
    - Пользователь выбирает действие "**Войти**"
    - Приложение отображает форму ввода данных: почту, пароль
    - Пользователь вводит данные в форму и завершает ввод данных
    - Система валидирует данные и возвращает токен авторизации
    - Приложение принимает ответ, запоминает токен и отображает успешность авторизации
    - Приложение проверяет наличие активной синхронизации на аккаунте: при наличии подгружает данные, при отсутствии предлагает включить синхронизацию.
- **Постусловия:**
    - У приложения есть авторизационный и обновляющий токен.
- **Альтернативные потоки:**
    - **А1**. Сервер вернул ответ о невалидных данных
        - Приложение уведомляет пользователя о некорректности какого-то поля и показывает требования к нему без ввода данных заново.
    - **А2**. Сервер вернул ответ о неверных данных (неверный пароль, аккаунта не существует -- эти случаи для пользователя неразличимы).
        - Приложение уведомляет пользователя о проблеме с данными и рекомендует действия по исправлению.
    - **А3**. Данные о финансах есть как на сервере, так и в приложении пользователя.
        - Приложение предлагает: оставить только данные устройства, совершить слияние, оставить только данные сервера, выполняет локальные преобразования данных в соответствии с выбором и отправляет на сервер актуальные данные.
- **Исключительные потоки:**
    - **E1**. Ошибка до момента отправления пакета (например, в формировании запроса).
    - **E2**. Запрос не дошел (плохая связь, сервер не работает).
        - Приложение уведомляет пользователя о работе в оффлайн режиме и предлагает создать локальную учетную запись с последующей привязкой к аккаунту (для **E1** и **E2**, для пользователя нет разницы в этих двух потоках)

### Подключение банковского аккаунта

- **ID**: UC-003
- **Название**: подключение интеграции
- **Акторы**: пользователь (основной), бекенд (вторичный), банковская система (вторичный)
- **Предусловия**: 
    - Приложение установлено.
    - У приложения есть доступ в интернет.
    - Пользователь авторизован с аккаунтом.
    - Синхронизация включена.
- **Основной поток:**
    - Пользователь выбирает действие "**Добавить банковский аккаунт**"
    - Приложение отображает форму выбора доступных банков.
    - Пользователь выбирает нужный банк.
    - Система перенаправляет пользователя на страницу авторизации банка.
    - Приложение принимает нужные данные от системы банка и сохраняет их.
    - Приложение обновляет данные пользователя локально.
    - Приложение проводит синхронизацию аккаунта
- **Постусловия:**
    - У пользователя есть новая банковская интеграця в приложении
- **Альтернативные потоки:**
    - **А1**. Банк отклонил подключение
        - Приложение уведомляет пользователя о неудаче, передает информацию от банка о причине, и предлагает рекомендации.
- **Исключительные потоки:**
    - **E1**. Ошибка на стороне приложения.
        - Приложение сообщает пользователю о неудаче и предлагает попробовать еще раз
    - **E2**. Ошибка на моменте подключения банка
        - Приложение уведомляет пользователя об ошибке и передает информацию об ошибке от банка (если есть), предлагая обратиться в поддержку банка.
    - **E3**. Ошибка на синхронизации с сервером
        - Приложение пробует еще раз с таймаутом. При невозможности через некоторое время сообщает пользователю о проблеме и предлагает обратиться в поддержку.
- **Бизнес-правила:**
    - **BR-002**. Аккаунт -- запись на сервере с уникальной почтой. Локальная учетная запись -- профиль пользователя на устройстве с или без привязки к аккаунту.

### Добавление транзакции

- **ID**: UC-004
- **Название**: Добавление транзакции
- **Акторы**: пользователь (основной), бекенд (вторичный)
- **Предусловия**: 
    - Приложение установлено.
    - Пользователь авторизован с аккаунтом или локальной учетной записью.
- **Основной поток:**
    - Пользователь выбирает действие "**Добавить транзакцию".**
    - Приложение позволяет выбрать пришедшую с сервера *неполную* транзакцию, либо создать новую вручную.
    - Приложение отображает форму сохранения транзакции.
    - Пользователь вводит сумму, счет (обязательно), название, категорию, бюджет, дату и время (опционально)
    - Система сохраняет транзакцию локально.
    - При наличии аккаунта проверяет новые транзакции на сервере и синхронизируется с сервером.
- **Постусловия:**
    - У пользователя есть новая транзакция в приложении (и на сервере -- при наличии аккаунта)
- **Альтернативные потоки:**
    - **А1**. Пользователь отменил транзакцию
        - Приложение уведомляет пользователя о неудаче и возвращается на предыдущий экран.
- **Исключительные потоки:**
    - **E1**. Ошибка синхронизации.
        - Приложение совершает 3 попытки и сообщает пользователю о неудаче.
